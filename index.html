Part 1 · The Problem

What is vibe coding?

"Just describe what you want and let the AI run with it."

Fast, fun, conversational — and increasingly how a lot of code gets written.

Define it honestly and without judgement. Vibe coding = prompt, accept, prompt again. No spec, no tests-first, no shared definition of done. It's genuinely great for prototypes, throwaway scripts, and learning. The problem is what happens when that style meets a real codebase, a team, and production.
🇮🇷 فارسی: صادقانه و بدون قضاوت تعریفش کنید. کدنویسی شهودی یعنی: پرامپت بده، بپذیر، دوباره پرامپت بده. نه مشخصاتی، نه تست‌محوری، نه تعریف مشترکی از «انجام‌شده» (Definition of Done). این روش برای نمونهٔ اولیه، اسکریپت یکبارمصرف و یادگیری عالی است. مشکل آنجاست که این سبک به یک کدبیس واقعی، یک تیم و محیط تولید (Production) برسد.

Vibe coding: the honest scorecard

✅ Where it shines

  • Prototypes & spikes
  • Learning a new API
  • One-off scripts
  • Exploring an idea fast

⚠️ Where it breaks

  • No shared definition of done
  • Untested, unverifiable output
  • Context lost between sessions
  • Architecture drift & rework
  • Doesn't scale across a team
The failure mode isn't "the AI writes bad code." It's that there is no artifact that says what "correct" means. So you can't verify, you can't onboard, you can't review at scale. Every session restarts from zero context. That's the gap SDD fills.
🇮🇷 فارسی: حالت شکست این نیست که «هوش مصنوعی کد بد می‌نویسد». مشکل این است که هیچ مصنوعی (Artifact) وجود ندارد که بگوید «درست» یعنی چه. پس نمی‌توان راستی‌آزمایی کرد، نمی‌توان افراد را وارد کار کرد، و نمی‌توان در مقیاس بازبینی کرد. هر جلسه از صفرِ زمینه شروع می‌شود. SDD دقیقاً همین شکاف را پر می‌کند.

The core gap

No spec → no shared truth →
no way to verify.

If you can't state the acceptance criteria, neither you nor the agent can prove the work is done.

This is the single most important slide of Part 1. Everything that follows is about creating a verifiable source of truth — the spec and its acceptance criteria — and then wiring agents to produce and check against it.
🇮🇷 فارسی: این مهم‌ترین اسلاید بخش ۱ است. هر چه در ادامه می‌آید دربارهٔ ساختن یک منبع حقیقتِ راستی‌آزمایی‌پذیر است — یعنی مشخصات و معیارهای پذیرش آن — و سپس سیم‌کشی عامل‌ها برای تولید و بررسی در برابر آن.
Part 2 · The Shift

Agentic engineering

AI agents doing real engineering work — under explicit guardrails, against an explicit spec, with automated verification.

Agentic engineering keeps the speed of AI but adds the discipline of engineering: explicit requirements, tests-first, verification gates, persistent project knowledge, and specialized advisors. The human moves from "typist" to "director / reviewer."
🇮🇷 فارسی: مهندسی عامل‌محور سرعت هوش مصنوعی را حفظ می‌کند ولی نظم مهندسی را اضافه می‌کند: الزامات صریح، تست‌محوری (TDD)، دروازه‌های راستی‌آزمایی، دانش پایدار پروژه و مشاوران تخصصی. نقش انسان از «تایپیست» به «کارگردان/بازبین» تغییر می‌کند.

🇮🇷 فارسی: سطر به سطر پیش بروید. ستون سمت چپ دقیقاً همان چارچوبی است که امروز می‌سازیم. هر سطر به یک مصنوع یا مهارت نگاشت می‌شود: PRD (brainstorm)، مشخصات+معیار پذیرش (write-spec)، TDD (implement-epic)، راستی‌آزمایی (verify-epic)، فایل‌های دانش و عامل‌های مشاور.

The spectrum, not a switch

Vibe→ Assisted→ Spec-driven→ Agentic

You don't abandon vibe coding — you graduate to it when the stakes are low, and lean on SDD when they're high.

Reassure the hands-on devs: vibe coding still has a place. The skill is knowing which mode the task deserves. SDD is the structure you reach for when code must live, ship, and be maintained by others.
🇮🇷 فارسی: به توسعه‌دهندگان عملی اطمینان بدهید: کدنویسی شهودی هنوز جایگاه دارد. مهارت این است که بدانی کار سزاوار کدام حالت است. SDD ساختاری است که وقتی کد باید زنده بماند، منتشر شود و توسط دیگران نگه‌داری شود، سراغش می‌روی.
Part 3 · The Method

Spec-Driven Development (SDD)

Specs are the program. Code is the compilation target. Agents do the compiling — and the checking.

The mental model: in SDD the *spec* is the durable artifact you maintain. Code is generated and regenerated against it. Acceptance criteria are the contract. This flips the usual "code is truth, docs rot" dynamic.
🇮🇷 فارسی: مدل ذهنی: در SDD «مشخصات» همان مصنوع پایداری است که نگه‌داری می‌کنی. کد در برابر آن تولید و بازتولید می‌شود. معیارهای پذیرش، قرارداد هستند. این، رابطهٔ معمول «کد حقیقت است و مستندات می‌پوسد» را وارونه می‌کند.

Four principles of SDD

  1. Specs before code. A PRD and technical spec exist before implementation.
  2. Acceptance Criteria are the contract. Every AC is testable and traceable.
  3. Tests first (TDD). Tests encode the ACs before the implementation does.
  4. Verify, don't trust. An automated gate proves each AC is met.
These four principles are the through-line of every skill we build. When in doubt, ask: "Does this move us toward a verifiable AC?" If not, it's vibe coding in disguise.
🇮🇷 فارسی: این چهار اصل، نخ تسبیح همهٔ مهارت‌هایی است که می‌سازیم. هرگاه شک کردی بپرس: «آیا این ما را به یک معیار پذیرشِ راستی‌آزمایی‌پذیر نزدیک می‌کند؟» اگر نه، این همان کدنویسی شهودی در لباس مبدل است.

Skill 1 of 5

🧠 brainstorm

Turn a fuzzy idea into an Epic PRD with a measurable confidence level.

PRD = Product Requirements Document

A concise, versioned document stating what to build and why — the problem, goals, requirements and acceptance criteria. An Epic PRD scopes one large feature (epic) and is the source of truth the whole SDD loop builds on.

brainstorm is where ambiguity goes to die. It produces the Epic PRD and drives a structured Q&A loop until the AI is ≥90% confident it understands the intent.
🇮🇷 فارسی: brainstorm جایی است که ابهام می‌میرد. این مهارت، PRD اپیک را تولید می‌کند و یک حلقهٔ ساختاریافتهٔ پرسش‌وپاسخ را پیش می‌برد تا هوش مصنوعی دست‌کم ۹۰٪ از درک منظور مطمئن شود.

brainstorm · workflow

  1. Create an Epic PRD if one doesn't exist.
  2. Put a confidence level in the PRD header.
  3. At the end, ask clarifying questions until confidence ≥ 90%.
  4. On re-run: reconcile answers into statements, requirements & ACs; remove resolved questions; keep an iteration log.
  5. Keep generating high-impact questions until ≥ 90%.
The magic is the loop + reconciliation. You never lose context: answered questions become durable requirements and ACs, the PRD's confidence climbs, and an iteration log records how you got there. This is the antidote to "context lost each session."
🇮🇷 فارسی: جادو در حلقه + تطبیق (Reconciliation) است. هرگز زمینه را از دست نمی‌دهی: پرسش‌های پاسخ‌داده‌شده به الزامات و معیارهای پذیرش پایدار تبدیل می‌شوند، اطمینان PRD بالا می‌رود و گزارش تکرار ثبت می‌کند که چطور به اینجا رسیدیم. این پادزهر «از دست رفتن زمینه» است.

The question format

### Q3. How should sessions expire? (confidence impact: HIGH)
- [ ] a) Never — sessions are permanent
- [ ] b) Fixed 24h TTL  **(Recommended)**
- [ ] c) Sliding window, refresh on activity
- [ ] d) Configurable per-tenant

> Mark your choice with an `x`, e.g. `- [x] b)`
  • Exactly 4 options.
  • One marked Recommended — but not pre-selected.
  • User answers by marking x.
Why this exact format? Four options forces the agent to think through real alternatives. "Recommended but not selected" gives guidance without removing your agency. The x-mark is trivially parseable on re-run. Consistency = automatability.
🇮🇷 فارسی: چرا دقیقاً همین قالب؟ چهار گزینه عامل را وادار می‌کند به بدیل‌های واقعی فکر کند. «پیشنهادشده ولی انتخاب‌نشده» راهنمایی می‌دهد بدون اینکه اختیار را از تو بگیرد. علامت x در اجرای مجدد به‌سادگی قابل‌تجزیه است. یکدستی یعنی خودکارسازی‌پذیری.

PRD header — confidence in the open

---
epic: checkout-redesign
confidence: 72%   # < 90% → more questions remain
status: drafting
iteration: 3
---

Confidence is a first-class, visible signal — not a vibe.

Making confidence explicit and numeric changes behavior. The team can see at a glance whether an epic is ready to spec. Under 90% means "keep asking." It also gives you a natural stopping rule instead of endless prompting.
🇮🇷 فارسی: آشکار و عددی‌کردن اطمینان، رفتار را تغییر می‌دهد. تیم در یک نگاه می‌بیند که آیا اپیک آمادهٔ نوشتن مشخصات هست یا نه. کمتر از ۹۰٪ یعنی «به پرسیدن ادامه بده». همچنین یک قانون توقف طبیعی به‌جای پرامپت‌زدن بی‌پایان می‌دهد.
Skill 2 of 5

📐 write-spec

Input: Epic Name. Output: a TDD-ready technical specification.

write-spec consumes the PRD and produces the engineering blueprint: architecture, design, testing strategy and an exact implementation plan — all traceable to ACs.
🇮🇷 فارسی: write-spec، PRD را مصرف می‌کند و نقشهٔ مهندسی را تولید می‌کند: معماری، طراحی، راهبرد تست و یک برنامهٔ پیاده‌سازی دقیق — همه قابل‌ردیابی به معیارهای پذیرش.

write-spec · workflow

  1. Read the Epic PRD.
  2. Create a detailed technical spec covering all requirements & ACs.
  3. Follow a TDD approach.
  4. Ensure every AC is covered by architecture · design · testing strategy · implementation plan.
  5. Produce an exact, step-by-step implementation plan for developers.
Key word: *exact*. The plan should be runnable by a developer (or implement-epic) without re-deciding architecture. Every AC must trace to a section of the spec — no orphan requirements, no untested ACs.
🇮🇷 فارسی: واژهٔ کلیدی: «دقیق». برنامه باید توسط یک توسعه‌دهنده (یا implement-epic) بدون تصمیم‌گیری مجدد دربارهٔ معماری قابل‌اجرا باشد. هر معیار پذیرش باید به بخشی از مشخصات ردیابی شود — نه الزام یتیم، نه معیار تست‌نشده.

Same loop — architecture decisions

  • At the end, ask architecture & technical-decision questions.
  • Each: 4 options, one Recommended (not selected).
  • On re-run: reconcile answers into architecture decisions, remove resolved questions, keep an iteration log.
  • Continue until confidence ≥ 90%.

Same proven mechanic as brainstorm — applied to technical choices.

Reuse the brainstorm pattern, now for technical decisions: datastore, sync vs async, API shape, etc. The reconciliation produces an Architecture Decision log inside the spec — your future ADRs, written as you go.
🇮🇷 فارسی: الگوی brainstorm را دوباره به کار می‌بریم، این بار برای تصمیم‌های فنی: پایگاه‌داده، همگام در برابر ناهمگام، شکل API و غیره. تطبیق، یک گزارش تصمیم معماری (ADR) درون مشخصات تولید می‌کند — یعنی ADRهای آینده‌ات، همان لحظه که پیش می‌روی نوشته می‌شوند.

🇮🇷 فارسی: این ماتریس قرارداد میان write-spec و verify-epic است. بعداً verify-epic دقیقاً همین سطرها را می‌پیماید و PASS/PARTIAL/FAIL گزارش می‌دهد. خانه‌های خالی = شکاف‌هایی که *پیش از* نوشتن کد می‌گیری.
Skill 3 of 5

⚙️ implement-epic

Input: Epic Name. Builds the epic TDD-style, then verifies it.

This is where code gets written — but in a disciplined order. Tests first, then functionality, then coverage validation, then fixing, then verification, then a report. Six steps.
🇮🇷 فارسی: اینجا کد نوشته می‌شود — اما با ترتیبی منظم. اول تست‌ها، سپس کارکرد، سپس اعتبارسنجی پوشش، سپس رفع خطا، سپس راستی‌آزمایی، و در پایان یک گزارش. شش گام.

implement-epic · 6 steps

1 · TestsWrite tests first, per spec & TDD
2 · FunctionalityBuild to requirements, ACs & spec
3 · Validate coverageEvery AC has impl + tests + passing run
4 · Fix failuresFix impl, re-run relevant tests, repeat
5 · VerifyRun verify-epic
6 · ReportFull implementation report
Step 4 loops until relevant tests pass — note "relevant," not the whole suite (speed). Step 5 hands off to the verification skill as the gate. Step 6 produces an auditable report. Walk each box briefly.
🇮🇷 فارسی: گام ۴ تا عبور تست‌های مرتبط تکرار می‌شود — توجه کنید «مرتبط»، نه کل مجموعه (برای سرعت). گام ۵ به مهارت راستی‌آزمایی به‌عنوان دروازه تحویل می‌دهد. گام ۶ یک گزارش قابل‌ممیزی تولید می‌کند. هر جعبه را کوتاه توضیح دهید.

The implementation report

  • ✅ Implemented requirements
  • 🎯 AC coverage
  • 🧪 Test results
  • ⚠️ Remaining risks
  • 📊 Confidence level

A reviewable artifact — not "trust me, it works."

The report is what a reviewer reads first. It turns "the agent did stuff" into a structured claim with evidence and an honest confidence number, including remaining risks. This is how you review agent work at scale.
🇮🇷 فارسی: گزارش، اولین چیزی است که بازبین می‌خواند. «عامل یک کارهایی کرد» را به یک ادعای ساختاریافته با شواهد و یک عدد اطمینان صادقانه — شامل ریسک‌های باقیمانده — تبدیل می‌کند. این‌طور می‌توان کار عامل را در مقیاس بازبینی کرد.
Skill 4 of 5

✅ verify-epic

Input: Epic Name. The independent gate that proves the work.

verify-epic is deliberately separate from implement-epic — separation of "doing" and "checking." It validates AC↔test coverage, runs the epic's tests, checks spec compliance, and emits a verdict per AC.
🇮🇷 فارسی: verify-epic عمداً از implement-epic جداست — جداسازی «انجام‌دادن» و «بررسی‌کردن». پوشش معیار↔تست را اعتبارسنجی می‌کند، تست‌های اپیک را اجرا می‌کند، انطباق با مشخصات را بررسی می‌کند و برای هر معیار یک حکم صادر می‌کند.

verify-epic · workflow

  1. Validate every AC has at least one test.
  2. Execute tests relevant to the Epic.
  3. Verify implementation coverage against ACs.
  4. Validate specification compliance.
  5. Produce a verification report.
Note step 1 is structural (does a test even exist for each AC?) before step 2 runs them. You can fail verification because an AC is *untested*, even if all existing tests pass. That closes a classic blind spot.
🇮🇷 فارسی: توجه کنید گام ۱ ساختاری است (آیا اصلاً برای هر معیار تستی وجود دارد؟) پیش از آنکه گام ۲ آن‌ها را اجرا کند. می‌توانی به‌دلیل *تست‌نشده‌بودن* یک معیار در راستی‌آزمایی مردود شوی، حتی اگر همهٔ تست‌های موجود سبز باشند. این یک نقطه‌کور کلاسیک را می‌بندد.

Report also lists: missing tests · missing implementation · risks · recommendations.

Three states, not two. PARTIAL is the honest middle that vibe coding hides. The report's recommendations feed straight back into implement-epic's fix loop. This is the verifiable "definition of done" we promised in Part 1.
🇮🇷 فارسی: سه حالت، نه دو حالت. PARTIAL همان میانهٔ صادقانه‌ای است که کدنویسی شهودی پنهانش می‌کند. توصیه‌های گزارش مستقیم به حلقهٔ رفع خطای implement-epic بازمی‌گردند. این همان «تعریف انجام‌شده»ی راستی‌آزمایی‌پذیری است که در بخش ۱ وعده دادیم.
Wiring it together

Verification integration

Two rules make verification fast during dev but thorough at the gate.
🇮🇷 فارسی: دو قاعده، راستی‌آزمایی را در حین توسعه سریع و سرِ دروازه دقیق می‌کند.

Two-speed verification

During implementation

  • Run only relevant tests
  • mvn -Dtest=AuthServiceTest test
  • No full-suite cost

Final verification

  • Complete epic validation — mvn verify
  • End-to-end verification
  • verify-epic is the last step of implement-epic
The tension: TDD needs fast loops, but a gate needs full confidence. Resolve it by scoping tests during the loop and going full + E2E only at the final gate. verify-epic is wired as implement-epic's terminal step so you can't "forget" to verify.
🇮🇷 فارسی: تنش: TDD به حلقه‌های سریع نیاز دارد، اما دروازه به اطمینان کامل. حلش با محدودکردن دامنهٔ تست‌ها در حلقه و رفتن به حالت کامل + سرتاسری فقط سرِ دروازهٔ نهایی است. verify-epic به‌عنوان گام پایانی implement-epic سیم‌کشی شده تا نتوانی راستی‌آزمایی را «فراموش» کنی.

🎭 Playwright integration

  1. Install Playwright MCP in the project.
  2. Configure E2E testing — tests with Playwright for Java alongside JUnit.
  3. Enable browser automation for E2E tests, verification workflows & user-journey validation.
  4. Make Playwright available to verify-epic.

Now verification can drive a real browser — not just unit tests.

Playwright MCP lets the verification agent actually click through the app and confirm user journeys. ACs about UX/flows become executable. This is how verify-epic graduates from "tests pass" to "the user journey works."
🇮🇷 فارسی: Playwright MCP اجازه می‌دهد عامل راستی‌آزمایی واقعاً در اپ کلیک کند و سفرهای کاربر را تأیید کند. معیارهای پذیرش مربوط به تجربهٔ کاربری/جریان‌ها قابل‌اجرا می‌شوند. این‌طور verify-epic از «تست‌ها پاس می‌شوند» به «سفر کاربر کار می‌کند» ارتقا می‌یابد.

Part 5 · Advisors

Advisory agents

Specialists that weigh in at the right moments.

Each advisor reads one reference file and participates across the whole lifecycle — brainstorming, spec, implementation, verification — surfacing domain recommendations when relevant decisions arise.
🇮🇷 فارسی: هر مشاور یک فایل مرجع را می‌خواند و در سراسر چرخهٔ حیات مشارکت می‌کند — طوفان فکری، مشخصات، پیاده‌سازی، راستی‌آزمایی — و هرگاه تصمیم مرتبطی پیش بیاید، توصیهٔ دامنه‌ای ارائه می‌دهد.

🇮🇷 فارسی: آن‌ها در هر چهار فاز مشارکت می‌کنند. مثل بازبین‌های ارشدی که همیشه در دسترس‌اند، هرگز خسته نمی‌شوند و همیشه به همان استاندارد مشترک ارجاع می‌دهند. این، تصمیم‌ها را میان اپیک‌ها و توسعه‌دهندگان یکدست نگه می‌دارد.

Part 6 · Documentation

📚 init-docs <domain>

Domain-driven documentation, generated from the code.

init-docs analyzes the codebase, identifies a domain, and generates living domain-driven docs. This becomes a primary knowledge source agents consult before any work.
🇮🇷 فارسی: init-docs کدبیس را تحلیل می‌کند، یک دامنه را شناسایی می‌کند و مستندات دامنه‌محورِ زنده تولید می‌کند. این به یک منبع دانش اصلی تبدیل می‌شود که عامل‌ها پیش از هر کاری به آن مراجعه می‌کنند.

init-docs · workflow

  1. Analyze the codebase.
  2. Identify the domain.
  3. Discover relevant code · APIs · tests · components.
  4. Generate domain-driven documentation.
docs/
├── domain-index.md        # navigation map + short domain descriptions
└── domain-<name>.md       # full domain documentation
Input is just a domain name. Output is a navigable index plus a detailed per-domain file. The index is the entry point agents use to orient before planning work.
🇮🇷 فارسی: ورودی فقط یک نام دامنه است. خروجی یک فهرست قابل‌پیمایش به‌علاوهٔ یک فایل تفصیلی برای هر دامنه است. فهرست، نقطهٔ ورودی است که عامل‌ها پیش از برنامه‌ریزی کار برای جهت‌یابی از آن استفاده می‌کنند.

Entities · Value Objects · Aggregates

Features

For each feature: description + acceptance criteria

This mirrors DDD: model, contracts, behaviors, components, and features with ACs. Note features carry ACs too — so documentation and the SDD loop speak the same language.
🇮🇷 فارسی: این، DDD را بازتاب می‌دهد: مدل، قراردادها، رفتارها، کامپوننت‌ها و قابلیت‌ها با معیارهای پذیرش. توجه کنید قابلیت‌ها هم معیار پذیرش دارند — پس مستندسازی و حلقهٔ SDD به یک زبان صحبت می‌کنند.

The final wiring step: reference the domain index from Copilot's instructions so every agent treats docs as authoritative, not optional. This closes the loop — knowledge flows into every phase of the lifecycle.

Part 7 · Live Demo

End-to-end in action

One epic, four skills, three advisors.

Demo time. If the live demo risks running long, narrate from these slides as a fallback. Pick a small, real epic so the loop completes inside ~8 minutes.

What just happened

  • An idea became a versioned, verifiable epic.
  • Every AC is traceable from PRD → spec → test → verdict.
  • Decisions were logged, not lost.

🇮🇷 فارسی: تضاد با بخش ۱ را برجسته کنید: نه بازبینی چشمی، نه زمینهٔ ازدست‌رفته، نه «درست به‌نظر می‌رسد». یک ردِ پایدار از قصد تا اثبات. این پاداش کل سامانه است.

Anti-patterns to avoid

Quick gut-check list so teams don't reinvent vibe coding with extra steps.
🇮🇷 فارسی: یک فهرست بررسی سریع تا تیم‌ها دوباره کدنویسی شهودی را با چند گام اضافه بازاختراع نکنند.

Watch out for…

Anti-patternFix
Spec theater — write spec, then ignore itverify-epic against the spec
ACs that can't be testedRewrite as observable, testable outcomes

Part 9 · Rollout

Adoption roadmap

Don't boil the ocean. Stage adoption so the team feels value early.
🇮🇷 فارسی: اقیانوس را به جوش نیاور. پذیرش را مرحله‌بندی کن تا تیم زود ارزش را حس کند.

Crawl → Walk → Run

The idea: adopt the framework in stages — don't boil the ocean. Each phase delivers value on its own, so the team feels wins early and builds momentum toward full agentic engineering.

Week 1 · CrawlAdd the 3 knowledge files + brainstorm. Use on one epic.
Week 2–3 · WalkAdd write-spec + implement-epic + verify-epic. TDD on a real feature.
Week 4 · RunAdvisors + Playwright + init-docs. Wire docs into instructions.
OngoingRefine references, grow domain docs, review reports in PRs.
Crawl proves value with minimal setup. Walk introduces the full TDD loop. Run adds the advisors, browser verification and living docs. Ongoing is where it compounds — every epic makes the knowledge base richer.
🇮🇷 فارسی: خزیدن با حداقل راه‌اندازی ارزش را اثبات می‌کند. راه‌رفتن حلقهٔ کامل TDD را معرفی می‌کند. دویدن مشاوران، راستی‌آزمایی مرورگری و مستندات زنده را اضافه می‌کند. «پیوسته» جایی است که اثرش انباشته می‌شود — هر اپیک پایگاه دانش را غنی‌تر می‌کند.

Success signals

  • 📉 Less rework & fewer "that's not what I meant" cycles.
  • 🔍 PR reviews start from a report, not a guess.
  • 🧪 Every AC has a test — coverage by design.
  • 🧠 New joiners ramp via knowledge + domain docs.
Give the team measurable outcomes to track. If these aren't improving, revisit the anti-patterns slide.
🇮🇷 فارسی: به تیم نتایج قابل‌سنجش بدهید تا دنبال کنند. اگر این‌ها بهتر نمی‌شوند، به اسلاید ضدالگوها برگردید.
